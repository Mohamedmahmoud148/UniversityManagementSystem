Architecture Style

The project uses a 3-Layer Clean Architecture variant:

Layer / ProjectResponsibilityDependencies
UniversityManagementSystem.CoreDomain Layer: Entities, Interfaces, DTOs, Exceptions, SettingsNone — pure C#
UniversityManagementSystem.InfrastructureApplication + Infrastructure: Services, EF Core, Jobs, Consumers, StorageCore only
UniversityManagementSystem.ApiPresentation: Controllers, Middleware, Filters, Program.csCore + Infrastructure
✅ Clean Architecture: Mostly Correct All service contracts live in Core/Interfaces. Infrastructure is fully swappable. Entities are pure domain objects with no EF annotations. Dependencies flow inward.
❌ One Real Violation UniversityManagementSystem.Core.csproj contains <FrameworkReference Include="Microsoft.AspNetCore.App" />. The Domain layer should never reference ASP.NET. This was added because IFormFile is used in service interfaces. Fix: replace IFormFile with a Stream + metadata DTO abstraction.

Request Lifecycle

  1. HTTP Request arrives at Railway load balancer
  2. ExceptionMiddleware wraps entire pipeline in try/catch
  3. Rate Limiter checks partition limits (global 1000/min, login 5/min, sensitive 10/min)
  4. Authentication Middleware validates JWT Bearer token
  5. Authorization checks [Authorize(Roles=...)]
  6. Controller action executes, calls Service layer
  7. Service interacts with AppDbContext (EF Core with global soft-delete filters)
  8. ResponseWrapperFilter wraps ObjectResult in ApiResponse<T>
  9. JSON Serializer converts ULID fields via UlidJsonConverterFactory
  10. Response returns with consistent wrapped JSON envelope

Scalability Audit

Capacity Analysis

ScaleVerdictBottleneckFix Required
1,000 users ✅ Handles easily No bottleneck at this scale None
10,000 users ✅ Handles well Single PostgreSQL instance begins to matter; no caching active Enable Redis data caching
100,000 users ⚠️ Needs work In-memory rate limiting fails horizontally; Hangfire PG backend under pressure Redis rate limiting, Hangfire Pro or Azure Service Bus, read replica
1,000,000 users ❌ Not ready Database sharding, event sourcing, full horizontal scaling required Full architecture re-evaluation

Key Scalability Bottlenecks

  1. In-memory rate limiting: PartitionedRateLimiter state lives in process memory. Two instances = double the allowed requests per user. Fix: AspNetCoreRateLimit with Redis storage.
  2. BCrypt in bulk import: 1,000 students × 100ms per hash = ~100 seconds blocking on CPU. Fix: Parallel.ForEachAsync with bounded concurrency.
  3. No read replicas: All reads hit primary PostgreSQL. AsNoTracking() reduces EF overhead but can't replace read scaling.
  4. Redis optional: System degrades gracefully when Redis is down — but then there's no caching at all. In production Redis should be required, not optional.
  5. Hangfire PostgreSQL: Excellent for moderate job volume. At 100K+ daily complaints, the job queue table becomes a hotspot. Consider migrating to Azure Service Bus at that scale.

Complete Project Documentation

1. Project Introduction

The University Management System (UMS) is a comprehensive academic management platform built as a graduation project. The system provides end-to-end management of the full university lifecycle — from institutional hierarchy management to student enrollment, AI-powered examinations, complaint intelligence, and automated grading. It is deployed on Railway cloud infrastructure using Docker containers and exposes a REST API consumed by web and mobile clients.

2. Problem Statement

Traditional university management relies heavily on manual processes: paper-based registration, Excel-based grading, verbal complaint handling, and disconnected administrative tools. These processes are error-prone, time-consuming, inaccessible remotely, unable to detect patterns in complaints or academic performance, and lacking in audit trails and accountability mechanisms.

3. Proposed Solution

A cloud-native, AI-integrated REST API system that manages the complete university hierarchy digitally, provides role-based access for all user types, integrates AI for intelligent exam generation and complaint analysis, automates grading and attendance, and provides real-time analytics and complaint clustering.

4. Objectives

  1. Digitize the complete university administrative hierarchy
  2. Provide secure, role-based access for Students, Doctors, Admins, and SuperAdmins
  3. Enable AI-powered academic assistance and exam generation
  4. Automate complaint analysis and priority escalation
  5. Support bulk student enrollment via Excel import
  6. Enable attendance tracking with QR codes
  7. Implement automated grading with GPA calculation
  8. Provide cloud file storage for course materials

Graduation Discussion Preparation

Architecture Questions

Q: Why did you choose Clean Architecture instead of a simple 3-layer MVC?

Clean Architecture ensures the domain logic (Core) is completely isolated from infrastructure concerns. If we wanted to switch from PostgreSQL to MongoDB, or from Hangfire to Azure Service Bus, only the Infrastructure layer changes — the Core and API layers remain untouched. This testability and independence is critical for long-term maintainability. It also enforces the Dependency Inversion Principle: high-level modules (Core) never depend on low-level modules (Infrastructure).

Q: Why ULID instead of GUID or integer IDs?

ULIDs are 26-character, URL-safe, and monotonically sortable. Unlike GUIDs, ULID insertion order matches creation time order, which prevents B-tree index fragmentation in PostgreSQL — improving insert and range-query performance. Unlike integers, ULIDs are unpredictable, preventing ID enumeration attacks (guessing other users' IDs by incrementing). The NUlid library provides full .NET integration with EF Core.

Q: Explain your soft delete implementation.

Every entity inherits from BaseEntity which has a DeletedAt nullable timestamp. In AppDbContext.OnModelCreating, we apply a global HasQueryFilter(x => x.DeletedAt == null) to every BaseEntity type. When SaveChangesAsync is called, we intercept EntityState.Deleted entries and convert them to Modified with DeletedAt = DateTime.UtcNow. This means deleted data is never physically removed — it's always auditable and recoverable.

Security Questions

Q: How does your authentication flow work end-to-end?

Login validates BCrypt password hash. On success, we generate a JWT (HS256, 24-hour expiry) with claims: nameid (SystemUser ID), ProfileId (role entity ID), role. We also generate a cryptographically random refresh token, store it in the database with expiry, and return both. The refresh endpoint validates the JWT signature even when expired, extracts the JTI, matches it to the stored token, marks it as used, and issues a new pair. The old refresh token cannot be reused.

Q: How do you prevent brute force attacks?

Three complementary layers: (1) Rate limiting — 5 login attempts per minute per IP address via ASP.NET Core rate limiter. (2) Account lockout — 5 failed password attempts triggers a 15-minute lockout stored in SystemUser.LockoutEnd, automatically reset on successful login. (3) BCrypt work factor — each password verification takes ~100ms by design, making parallel brute force computationally infeasible.

Q: What is MustChangePassword and why did you implement it?

When an admin creates any new user, their initial password is "123456" — a known default. MustChangePassword = true is set on account creation. The login response includes RequiresPasswordChange: true, signaling the frontend to redirect to the change-password screen before allowing access to any feature. After the user changes their password, MustChangePassword = false is persisted. This ensures compromised accounts from default passwords are immediately forced to change credentials.

Database Questions

Q: Why PostgreSQL instead of SQL Server or MySQL?

PostgreSQL offers superior features for this use case: the citext extension for case-insensitive email columns with proper indexing; native JSONB for storing exam answers; excellent EF Core integration; open-source with no licensing costs; strong Railway/cloud support; and better performance on complex joins and write-heavy workloads.

Q: How do you handle concurrent exam submissions (two tabs, race condition)?

The ExamSubmission table has a unique composite index on (ExamId, StudentId). If a student attempts to submit twice simultaneously, the second INSERT throws a DbUpdateException (unique constraint violation) at the database level — before any application code can process it. This is more reliable than application-level checks alone, which can have race conditions when two requests run in parallel.

Q: What is the purpose of the AcademicYearDepartment junction table?

A department runs different subjects in different academic years. This junction table maps which departments are active in which academic years, allowing the system to track which subjects and offerings belong to which academic year per department. Without it, you couldn't model that Department A offers subjects X, Y in Year 1 but subjects A, B in Year 2 — everything would be flat with no temporal separation.

AI Integration Questions

Q: How does the AI complaint analysis work end-to-end?

When a student files a complaint, ComplaintService persists it to the database and immediately enqueues a Hangfire background job ProcessNewComplaintAsync(complaintId). The background job loads the complaint, sends it to the FastAPI AI service at /analyze-complaint. The AI returns sentiment score (-1 to +1), category, severity (normal/high/critical), a summary, and optionally a DuplicateGroupId for clustering similar complaints. The job saves a ComplaintAnalysis record, elevates the complaint priority if severity is critical or high, and creates or updates a ComplaintCluster for trending detection.

Q: Why is the AI service a separate FastAPI service instead of being inside .NET?

Python has far superior AI/ML ecosystem support — LangChain, HuggingFace, OpenAI SDK, spaCy are Python-native and production-tested. Keeping AI in FastAPI also allows independent scaling (GPU instances for AI without affecting the main API), independent deployment (update AI models without redeploying the entire .NET application), and using Python-specific AI libraries. The .NET backend uses Polly with circuit breaker and retry so if the AI service is temporarily down, the main system continues operating normally.
